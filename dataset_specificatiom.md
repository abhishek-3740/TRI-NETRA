# TRINETRA - Data Generation Specification

## Purpose

Since Surat Police has not provided real investigation datasets, we will generate our own datasets that closely resemble real-world banking and telecom records.

The goal is NOT to create fake fraud cases.

The goal is to simulate a realistic environment where thousands of normal users exist, a few entities naturally overlap across different organizations, and only a very small number of entities exhibit statistically unusual behaviour.

The analysis engine should discover those unusual entities automatically.

---

# Core Philosophy

The project contains three independent datasets.

1. Bank Dataset
2. CDR Dataset
3. IPDR Dataset

These datasets should NOT be generated by copying one another.

Instead, think of them as data coming from three completely different organizations.

Bank
↓

Stores banking information.

Telecom Operator
↓

Stores phone call information.

Internet Service Provider
↓

Stores internet session information.

Each organization only knows its own data.

The investigation platform is responsible for discovering relationships between them.

---

# Fundamental Assumptions

The system should assume that

• Most users are normal.

• Most transactions are legitimate.

• Most calls are unrelated.

• Most internet sessions are normal.

Only a very small percentage of records should eventually appear suspicious.

This closely resembles real-world investigations.

Example

100000 Bank Transactions

↓

99850 Normal

↓

150 Potentially Suspicious

Similarly,

200000 CDR Records

↓

199000 Normal

↓

1000 Related to Bank Entities

↓

100 Potentially Suspicious

Again,

500000 IPDR Sessions

↓

Almost all normal

↓

Small percentage connected to suspicious entities.

The system should never know these beforehand.

# Dataset Independence

Every dataset should be independently generated.

The CDR dataset should never be created by copying the bank dataset.

Similarly,

IPDR should never copy CDR.

Each dataset should be realistic on its own.

Only later should some entities naturally overlap.

# Noise

Noise is essential.

Most records should never become useful during investigation.

Large amounts of unrelated data should exist.

This forces the analysis engine to search instead of simply matching everything.

---

# Suspicious Behaviour Simulation

Do NOT generate "fraud".

Instead,

simulate statistically uncommon behaviour.

Examples

Very high transaction frequency.

Money rapidly moving across multiple accounts.

Many unknown beneficiaries.

Repeated calls before transactions.

Same device used by multiple accounts.

Large transactions outside normal behaviour.

Unusual login times.

Unusual locations.

Multiple bank accounts sharing devices.

Circular money movement.

Frequent account creation followed by high-value transfers.

The generator should never label these as fraud.

They simply represent unusual behavioural patterns.

The investigation engine decides whether they are suspicious.

---

# Missing Data

Real datasets contain missing values.

Some locations should be unavailable.

Some device information should be missing.

Some tower IDs may be absent.

Some records may be incomplete.

Perfect datasets should never be generated.

---

# Data Distribution

Every column should follow realistic distributions.

Most transactions should be low value.

Large transactions should be rare.

Most calls should be short.

Very long calls should be uncommon.

Most users should have predictable daily routines.

Only a few should significantly deviate.

---

# Dataset Validation

Before exporting,

validate

Timestamp consistency.

Location consistency.

Phone consistency.

Device consistency.

Relationship consistency.

Duplicate handling.

Missing value percentages.

Distribution correctness.

Behaviour realism.

---

# Final Goal

The generated datasets should satisfy the following conditions.

Each dataset should appear to originate from a different organization.

Each dataset should be independently meaningful.

Relationships should exist but remain hidden.

Most data should be normal.

Only a very small percentage of entities should exhibit unusual behaviour.

The investigation platform should discover these relationships using fusion, graph analysis, rule engines, and machine learning.

The datasets themselves should never reveal the answer.
